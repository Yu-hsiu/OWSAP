## OWASP API Security Top 10(2023)逐項整理

### API1:2023 物件層級授權失效(BOLA, Broken Object Level Authorization)
- **風險**:API 未確實驗證「這個使用者有沒有權限存取這個物件」,攻擊者換掉資源
  ID 就能讀到別人的資料。BOLA 關注的是對「資料」的存取。
- **情境**:使用者用 `GET /api/v3/users?id=2727` 取自己資料,把 ID 改成 2728
  就拿到別人的資訊。
- **防禦**:每個用用戶端輸入去查資料庫的函數都要做授權檢查(RBAC + 使用者原則);
  資源 ID 改用不可預測的 GUID/UUID;寫專門測授權的自動化測試,測試沒過就不部署。

### API2:2023 認證失效(Broken Authentication)
- **風險**:認證流程的任何弱點——弱令牌、可暴力破解、密碼儲存不當等。
- **情境**:密碼重置要輸入 email + 6 位數驗證碼,但 API 不限次數,攻擊者送 100
  萬次請求即可猜中並重置任何人的密碼。
- **防禦**:不自幹認證/令牌/密碼儲存,用成熟標準;認證端點加速率限制、帳號鎖定、
  CAPTCHA、MFA;變更密碼/email/MFA 等敏感操作要求重新驗證身分。

### API3:2023 物件屬性層級授權失效(BOPLA, Broken Object Property Level Authorization)
- **風險**:2019 版「過度資料暴露 + 批量賦值」的合併。端點回傳了不該讀的屬性,
  或允許使用者改/加不該碰的屬性。
- **情境**:過度暴露——查自己帳戶卻回傳全部人的資料;批量賦值——建帳號請求偷塞
  `{"isAdmin": true}`,後端沒過濾就升成管理員。
- **防禦**:別用 `to_json()`/`to_string()` 通用序列化,手動挑要回傳的屬性;
  別用把輸入自動綁到內部物件的函數,只允許該更新的欄位;用 schema 驗證回傳內容,
  資料結構保持在滿足業務的最低限度。

### API4:2023 資源消耗不受限(Unrestricted Resource Consumption)
- **風險**:缺乏防止過度使用的保護,可導致 DoS 或雲端費用暴增。
- **情境**:沒限制單次請求回傳筆數、執行超時、最大記憶體、第三方服務花費上限。
- **防禦**:對呼叫次數做速率限制,用 Docker 等限制記憶體/CPU/程序數;伺服器端
  驗證查詢字串與 body 參數(尤其控制回傳筆數的);限制所有輸入的大小上限。

### API5:2023 功能層級授權失效(BFLA, Broken Function Level Authorization)
- **風險**:對「功能」的存取控制不足。BOLA 管資料存取,BFLA 管「修改/刪除資料」
  或執行別的角色(如管理動作)的功能,低權限使用者跑了高權限操作。
- **情境**:一般使用者用 `/{userid}/account/balance`,管理員用
  `/admin/account/{userid}`;若一般使用者能成功呼叫管理員端點就是 BFLA。
- **防禦**:授權模組預設拒絕、只對特定角色明確放行;所有管理端點繼承有做角色檢查
  的抽象控制器;定期依業務邏輯與群組階層檢視端點。

### API6:2023 敏感業務流程存取不受限(Unrestricted Access to Sensitive Business Flows)
- **風險**:攻擊者用自動化濫用某個業務流程,造成業務損害(而非技術漏洞本身)。
- **情境**:黃牛用機器人在極短時間反覆下單,秒清 PS5 庫存,讓一般人買不到。
- **防禦**:業務層先找出可能被濫用的流程,工程層再選保護手段(裝置指紋、CAPTCHA
  或生物辨識做真人檢測、分析非人類操作模式);必要時封鎖 Tor 出口節點與已知代理 IP。

### API7:2023 伺服器端請求偽造(SSRF, Server-Side Request Forgery)
- **風險**:使用者能控制伺服器要去抓的遠端資源,誘使伺服器對攻擊者指定目標發請求。
- **情境**:更新庫存的請求裡帶一個 URL,攻擊者換成指向 localhost 的密碼檔路徑,
  伺服器就把檔案內容回傳給攻擊者。
- **防禦**:對來源、URL scheme、連接埠、媒體類型做白名單;隔離負責抓資源的機制、
  停用 HTTP 重導向;用成熟的 URL 解析器並淨化所有輸入。

### API8:2023 安全設定錯誤(Security Misconfiguration)
- **風險**:大雜燴類別——錯誤標頭、預設帳號、開放不必要的 HTTP 方法、缺輸入淨化、
  洩漏詳細錯誤訊息等。
- **情境**:用 `X-Response-Time` 標頭做時序攻擊,暴力枚舉帳號編號,回應時間變長
  代表該帳號存在。
- **防禦**:可重複的環境硬化流程 + 自動化持續評估設定;全程加密通訊、只開必要的
  HTTP 動詞;強制回應(含錯誤回應)符合 schema,避免洩漏例外堆疊等系統細節。

### API9:2023 庫存管理不當(Improper Inventory Management)
- **風險**:暴露了非生產環境或不再維護的舊版 API,舊版少了修補與安全標準,成為
  攻其他漏洞的入口。
- **情境**:攻擊者從舊文件或猜測找到還活著的 `/v1/`、`/test/` 路徑,利用未修補
  版本存取行政功能或拿敏感資料。
- **防禦**:盤點所有 API 主機並記錄環境/版本/存取對象;用開放標準自動生成文件並
  納入 CI/CD;非生產環境別用生產資料,對所有暴露版本(不只當前生產版)用 API 防火牆。

### API10:2023 不安全地取用 API(Unsafe Consumption of APIs)
- **風險**:關注 API「取用者」——開發者過度信任第三方 API 回來的資料,沒像對待
  使用者輸入那樣驗證與淨化。
- **情境**:被取用的第三方 API 被攻擊者控制且連線未加密,攻擊者可攔截敏感資料,
  或透過該連線對取用方發動注入攻擊。
- **防禦**:把第三方資料當成跟使用者輸入同等風險,用前先驗證淨化;評估供應商時
  審查其 API 安全狀態、確保加密通道;維護已知安全的重導向清單,別盲從第三方的
  重導向指令。

## 習得能力總結

1. 能說出 OWASP API Security Top 10(2023)全部十項的官方編號、名稱與定義,並
   分辨易混淆的三組:BOLA(資料存取)vs BFLA(功能存取)、BOPLA 涵蓋的過度資料
   暴露 vs 批量賦值。
2. 能針對每一項風險,舉出一個具體攻擊情境,並對應提出至少一項防禦措施(如授權
   檢查、速率限制、schema 驗證、白名單、環境硬化)。
3. 能辨識 API 設計/程式碼中的常見弱點徵兆(可預測 ID、通用序列化、無速率限制、
   使用者可控 URL、暴露舊版路徑),作為 code review 或威脅建模的檢查清單。
4. 能說明 API 安全掃描與分析工具(Burp Suite、OWASP ZAP、Wireshark、Nikto 等)
   各自的用途類別,作為後續動手實作的選型基礎。
5. 具備 CASA 認證方向所需的 API 安全風險知識框架,可作為進一步 hands-on 滲透
   測試學習的理論地基。
